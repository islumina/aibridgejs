# aibridgejs

> 不依賴傳輸層的橋接核心，支援 iframe、Flutter InAppWebView 與 mock runtime。採用 JSON 安全的請求/回應協定，內建 AbortSignal 取消機制，並透過適配器隔離宿主耦合。隸屬 ai*js micro-runtime 生態系。目標打包大小：core gzip < 10 KB。

## 概述

aibridgejs 解決混合應用的宿主橋接問題。不同的 runtime 各有不同的橋接語意：iframe 使用 postMessage，Flutter InAppWebView 暴露具名的 JS handler 並觸發平台就緒事件，本機開發環境則兩者都不需要。aibridgejs 透過可替換的適配器，在三者之上提供同一套穩定的 API。

核心套件是零執行期依賴的純 TypeScript。所有宿主特定邏輯封裝在適配器中，以同一個 npm 套件的子路徑匯出形式發佈，按需載入。所有非同步公開方法接受 AbortSignal 進行取消。長生命週期物件遵循 createX() / dispose() 生命週期合約。套件設計為從應用層使用；核心不從 svelte、pixi.js 或任何宿主全域變數 import。

請求/回應的對應關係透過每個封包中嵌入的字串 id 實現。並行的 call() 呼叫各自取得唯一的 id；回應依據 id 匹配回對應的待處理項目後即移除。就緒閘控確保在適配器通知宿主已就緒之前不會發出任何呼叫。

## 核心 API

`createBridge({ adapter, timeoutMs? })` 回傳一個 Bridge，包含：

- `ready(options?)` — Promise<void>。當宿主就緒時 resolve。所有 call() 與 emit() 內部會等待此 Promise。接受 AbortSignal。
- `call(method, payload?, options?)` — Promise<unknown>。送出 request 封包並以 response payload resolve。在宿主錯誤、逾時、中止或 dispose 時 reject。接受個別呼叫的 timeoutMs 與 AbortSignal。
- `emit(event, payload?)` — Promise<void>。向宿主送出 event 封包。發出即忘，不追蹤任何回應。
- `on(event, listener, options?)` — () => void。訂閱入站的宿主事件。選項：once、signal。回傳取消訂閱函式。
- `platform()` — string。回傳適配器的平台識別字：'iframe' | 'flutter' | 'mock' | 'cocos' | 'unknown'。
- `reset()` — void。清除待處理呼叫、監聽器與就緒狀態，但不釋放適配器。
- `dispose()` — void。以 BridgeDisposedError reject 所有待處理呼叫，移除所有監聽器，並呼叫 adapter.dispose()。

## 傳輸協定

所有訊息都是 BridgeEnvelope — 以 kind 欄位為判別子的聯合型別：

- kind 'request'：id（string）、method（string）、payload?（unknown）、timestamp（number）
- kind 'response'：id（string，回應對應的 request id）、ok（boolean）、payload?（unknown）、error?（{ code, message, detail? }）、timestamp（number）
- kind 'event'：event（string）、payload?（unknown）、timestamp（number）

所有 payload 必須是 JSON 可序列化的資料。類別實例、Date、Map、Set 與 undefined 均不合法。適配器不得增減欄位。

## 安全性

iframe 適配器：targetOrigin 必須是精確的 HTTPS origin——萬用字元 * 在建構時即拋出例外。每一則入站訊息都會驗證 event.origin 與 event.source，來源錯誤的訊息會被靜默丟棄。

Flutter 適配器：在原生端的 WebMessageListener 設定 allowedOriginRules；無子 frame 時將 handler 限制為主 frame 專用；使用 InAppWebView user scripts 初始化 bridge，而非在頁面載入後呼叫 evaluateJavascript；絕不記錄可能包含 session token 的 BridgeEnvelope payload。

核心：任何無法解析為合法 BridgeEnvelope 的入站訊息會被靜默丟棄。核心本身不輸出任何日誌。

## TDD 驗收條件

任何正確的實作必須滿足以下全部七項：

1. 就緒閘控 — call() 與 emit() 在 ready() resolve 前排隊等候。
2. ID 對應 — 並行的 call() 呼叫各自依照回應的 id 獨立 resolve。
3. 逾時 reject — 待處理呼叫在 timeoutMs 後 reject，並移除待處理項目。
4. 中止 reject — AbortSignal 在呼叫進行中觸發，呼叫 reject，並移除待處理項目。
5. 格式錯誤丟棄 — 無法解析為 BridgeEnvelope 的入站訊息被靜默丟棄。
6. 來源拒絕 — 來自錯誤 origin 或 source 的 iframe 訊息被靜默丟棄。
7. Dispose reject — dispose() 以 BridgeDisposedError reject 所有進行中的呼叫。
